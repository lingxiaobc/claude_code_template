# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在本仓库中工作时提供指导。

## 项目概述

这是一个 Claude Code 配置模板仓库，用于快速初始化新项目的 AI 辅助开发环境。仓库本身不含业务代码，而是提供标准化的 Claude Code 配置基础设施。

## 架构

```
.claude/
├── agents/           # 自定义专用代理
│   ├── project-searcher.md  # 项目内容搜索代理（强制代理，禁止主程序绕过）
│   └── web-searcher.md      # 联网搜索代理（强制代理，禁止主程序绕过）
├── hooks/            # 自动化钩子
│   └── auto-commit.sh       # PostToolUse 自动提交脚本
├── skills/           # 自定义技能
│   └── plan-tasks/          # 大规模变更任务规划技能
├── settings.json     # 主配置（钩子、共享设置）
└── settings.local.json     # 本地覆盖（API 密钥、模型配置，已在 .gitignore 中）
```

**核心机制：**
- **强制代理模式**：项目搜索必须通过 `project-searcher`，网络搜索必须通过 `web-searcher`，主程序不得直接使用 Grep/Glob/Read 或 WebSearch/WebFetch 或通过 Bash 命令（curl、wget、find、grep 等）绕过
- **代理模型**：代理默认使用 haiku 模型，如不可用需修改代理 `.md` 文件中的 `model` 字段
- **自动提交**：每次 Edit/Write/NotebookEdit 操作后，`auto-commit.sh` 自动暂存并提交变更（跳过 `.git/` 和 `settings.local.json`，跳过 >10MB 文件）
- **提交格式**：`auto: [update|create] [文件名]`

## 行为准则

以下准则旨在减少常见 LLM 编码错误。偏向谨慎而非速度，简单任务可灵活处理。

### 1. 编码前先思考

**不要假设，不要隐藏困惑，主动暴露权衡。**

- 明确陈述假设，不确定就提问
- 存在多种理解时，全部列出——不要默默选择
- 有更简单的方案就说出来，必要时反驳
- 不清楚就停下来，说明困惑点，然后提问

### 2. 简单优先

**用最少的代码解决问题，不做推测性设计。**

- 不添加未被要求的功能
- 不为单次使用的代码创建抽象
- 不添加未被请求的"灵活性"或"可配置性"
- 不为不可能发生的场景写错误处理
- 如果写了 200 行但 50 行就能解决，重写

### 3. 精准修改

**只改必须改的，只清理自己造成的混乱。**

- 不"改进"相邻代码、注释或格式
- 不重构没坏的东西
- 匹配已有风格，即使自己会写得更不同
- 发现无关的死代码时提一下——不要删除

自己的变更产生了废弃代码时：
- 删除因你的变更而变得无用的 import/变量/函数
- 不要删除之前就存在的死代码（除非被要求）

**检验标准：每行变更都应能追溯到用户的请求。**

### 4. 目标驱动执行

**定义成功标准，循环直至验证通过。**

将任务转化为可验证的目标：
- "添加校验" → "为无效输入写测试，然后让测试通过"
- "修复 bug" → "写一个能复现的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务应给出简要计划（变更涉及多个文件或模块时，使用 `plan-tasks` 技能制定结构化任务明细）：
```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

**准则有效的标志：** diff 中不必要变更更少，过度复杂导致的重写更少，澄清问题在实现之前而非犯错之后提出。

### 5. 不确定时必须验证

**禁止用猜测代替事实。所有不确定的陈述必须经过验证。**

当你的输出中即将出现以下词语时，说明你并不确定——此时必须停下验证：
- "可能"、"大概"、"也许"、"似乎"、"应该是"、"估计"
- "让我尝试一下"、"试试看这个方法"
- 或任何其他表达不确定性的措辞

**验证方式（按场景选择）：**

- **项目内部事实**（包管理器、文件格式、目录结构、依赖版本等）→ 检测项目实际状态（读文件、执行检测命令），不要猜
- **外部技术事实**（框架用法、配置语法、API 行为等）→ 联网搜索官方文档，确认后再说
- **两种都不确定时** → 两件事都做

**反例：**
- ❌ "用户可能用的是 npm" → ✅ 检查 `package-lock.json`、`node_modules`、`package.json` 等，确定后给出结论
- ❌ "Claude Code 的技能配置文件可能为 yaml 格式" → ✅ 搜索官方文档，确认实际格式后回答
- ❌ "这个报错大概是因为版本不兼容" → ✅ 查看实际版本号、搜索兼容性矩阵，给出确定结论

### 6. 主程序委托执行

**主程序是调度者和审核者，不是执行者。**

- 主程序**不直接执行任务**（不直接编写代码、搜索项目、查询网络），所有执行工作交给子代理完成
- 主程序的职责：**审核、诊断、判断、优化**子代理的执行过程，严格把控方向，确保子代理编写代码时不偏方向、不过度复杂化
- 主程序根据 `TASK/` 中的**计划文件和任务文件**内容，优化、调整子代理的执行方向并给出回复，确保整个流程和步骤不偏
- 子代理可以使用 `.claude/skills/` 目录下的技能，但主程序需根据子代理的**职责和功能**开放对应的技能权限，不相关的技能不授予
- 子代理执行完毕后，主程序审核其产出，发现问题则退回修正或追加指导

**主程序专属技能（禁止委托给子代理）：**

以下技能只能由主程序直接调用，子代理不得使用：
- `plan-tasks` — 大规模变更任务规划，属于主程序的调度核心能力
- `skill-creator` — 技能创建与管理，属于主程序的元管理能力

**任务并行与协作策略：**

- 主程序根据 `TASK/` 目录中设定的计划，分析任务之间的依赖关系
- **彼此独立的任务** → 并行开多个子代理同时执行，互不阻塞
- **存在数据交流或信息传递的任务** → 开启多团体协作（Team），让相关子代理在同一个任务列表下协调工作
- 典型场景：制定方案、规划架构等需要多代理信息共享的任务，使用 `TeamCreate` 建立协作组

**执行流程：**
```
1. 读取 TASK/ 中的计划和任务文件，分析依赖关系
2. 根据子代理职责分配技能权限，独立任务并行分发
3. 将有依赖/交流的任务组织为协作组
4. 基于计划文件内容持续审核子代理产出，把控方向和复杂度
5. 发现偏差时及时干预：退回修正或补充指导
```
